# v6.0 — MapperFactoryBean 代理：全链路打通

## 1. 版本目标

解决"如何让分片逻辑对业务代码完全透明"的问题。

实现 `ShardingMapperFactoryBean`，用 JDK 动态代理包装 MyBatis Mapper，在方法调用前自动提取 BoundSql 参数、调用 Engine 计算分片结果、写入 ShardingContext，完成 v1.0~v5.0 所有组件的串联。业务代码只需正常调用 `userMapper.insert(user)`，框架自动完成路由和 SQL 改写。

---

## 2. 核心功能说明

### 类比

> 在 Mapper 方法的"门口"加一个保安，每次调用前先算好"去哪个仓库哪个货架"，写在便利贴（ThreadLocal）上，然后放行。后面的审查员（拦截器）和仓库管理员（DataSource）各司其职，整条流水线自动运转。

### 完整链路流程图

```
userMapper.insert(new User(3, "ma", 20))
        │  (JDK Proxy 拦截)
        ▼
ShardingMapperFactoryBean 内部 InvocationHandler
  1. mapperId = "UserMapper.insert"
  2. MappedStatement → BoundSql → sql = "insert into user..."
  3. getParams(boundSql, args) → Object[]{3, "ma", 20}
  4. engine.sharding(sql, params) → ShardingResult("ds1", "insert into user0...")
  5. ShardingContext.set(result)
        │
        ▼
method.invoke(originalProxy, args)  ← 触发真实 MyBatis 执行
        │
        ▼
SqlStatementInterceptor              ← 替换 BoundSql.sql 为 "insert into user0..."
        │
        ▼
ShardingDataSource                   ← determineCurrentLookupKey() = "ds1"
        │
        ▼
ds1 数据库执行 "insert into user0(id,name,age) values(3,'ma',20)"
```

### 各组件职责分工

```
┌─────────────────────────────────────────────────────────┐
│  ShardingMapperFactoryBean (代理层)                      │
│  职责：提取参数 → 调用 Engine → 写入 ShardingContext      │
├─────────────────────────────────────────────────────────┤
│  SqlStatementInterceptor (MyBatis 拦截层)                │
│  职责：从 ShardingContext 读取目标 SQL → 替换 BoundSql    │
├─────────────────────────────────────────────────────────┤
│  ShardingDataSource (连接层)                             │
│  职责：从 ShardingContext 读取目标 DS → 路由连接          │
└─────────────────────────────────────────────────────────┘
```

## 4. 代码实现说明

### ShardingMapperFactoryBean.java

继承 MyBatis 的 `MapperFactoryBean`，重写 `getObject()` 返回带分片逻辑的 JDK 动态代理。

```java
public class ShardingMapperFactoryBean<T> extends MapperFactoryBean<T> {

    @Autowired
    private ShardingEngine shardingEngine;

    @Autowired
    private SqlSessionFactory sqlSessionFactory;

    public ShardingMapperFactoryBean() {}

    public ShardingMapperFactoryBean(Class<T> mapperInterface) {
        super(mapperInterface);
    }

    @Override
    @SuppressWarnings("unchecked")
    public T getObject() throws Exception {
        // 获取 MyBatis 原始 Mapper 代理
        T originalMapper = super.getObject();
        Class<T> mapperInterface = getObjectType();

        // 用 JDK 动态代理再包一层，注入分片逻辑
        return (T) Proxy.newProxyInstance(
            mapperInterface.getClassLoader(),
            new Class[]{mapperInterface},
            new ShardingInvocationHandler(originalMapper, mapperInterface)
        );
    }

    /**
     * 分片逻辑的核心：InvocationHandler
     */
    private class ShardingInvocationHandler implements InvocationHandler {

        private final T originalMapper;
        private final Class<T> mapperInterface;

        ShardingInvocationHandler(T originalMapper, Class<T> mapperInterface) {
            this.originalMapper = originalMapper;
            this.mapperInterface = mapperInterface;
        }

        @Override
        public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
            // Object 方法（equals/hashCode/toString）直接透传
            if (Object.class.equals(method.getDeclaringClass())) {
                return method.invoke(originalMapper, args);
            }

            // 构造 MappedStatement ID：全限定类名.方法名
            String mapperId = mapperInterface.getName() + "." + method.getName();

            // 从 MyBatis Configuration 获取 MappedStatement
            Configuration configuration = sqlSessionFactory.getConfiguration();
            if (!configuration.hasStatement(mapperId)) {
                return method.invoke(originalMapper, args);
            }

            MappedStatement ms = configuration.getMappedStatement(mapperId);

            // 构造参数对象（MyBatis 的参数封装逻辑）
            Object paramObject = buildParamObject(ms, args);

            // 获取 BoundSql（含原始逻辑 SQL）
            BoundSql boundSql = ms.getBoundSql(paramObject);
            String sql = boundSql.getSql().trim();

            // 将参数展开为 Object 数组，供 Engine 解析
            Object[] flatArgs = flattenArgs(boundSql, paramObject);

            // 调用分片引擎计算路由结果
            ShardingResult result = shardingEngine.sharding(sql, flatArgs);

            // 写入 ThreadLocal，供后续拦截器和 DataSource 使用
            ShardingContext.set(result);

            try {
                return method.invoke(originalMapper, args);
            } finally {
                // v7.0 会在此处添加 ShardingContext.remove()
            }
        }

        /**
         * 将 MyBatis 的参数对象展开为有序的 Object 数组
         * 供 StandardShardingEngine 按位置与 SQL 中的 ? 对齐
         */
        private Object[] flattenArgs(BoundSql boundSql, Object paramObject) {
            List<ParameterMapping> mappings = boundSql.getParameterMappings();
            if (mappings == null || mappings.isEmpty()) {
                return new Object[0];
            }

            MetaObject metaParam = SystemMetaObject.forObject(paramObject);
            Object[] result = new Object[mappings.size()];
            for (int i = 0; i < mappings.size(); i++) {
                String propName = mappings.get(i).getProperty();
                if (metaParam.hasGetter(propName)) {
                    result[i] = metaParam.getValue(propName);
                } else if (boundSql.hasAdditionalParameter(propName)) {
                    result[i] = boundSql.getAdditionalParameter(propName);
                }
            }
            return result;
        }

        /**
         * 模拟 MyBatis DefaultParameterHandler 的参数封装逻辑
         */
        private Object buildParamObject(MappedStatement ms, Object[] args) {
            if (args == null || args.length == 0) return null;
            if (args.length == 1) return args[0];

            // 多参数时封装为 Map（key = param1, param2, ... 或 @Param 注解值）
            ParamNameResolver resolver = new ParamNameResolver(
                ms.getConfiguration(), ms.getStatementType().getClass().getMethod(""));
            // 简化处理：直接返回第一个参数（实际项目中需完整实现）
            return args[0];
        }
    }
}
```

```java
@SpringBootApplication

    basePackages = "com.example.masharding.mapper",
    factoryBean = ShardingMapperFactoryBean.class  // 关键：指定自定义工厂
```

## 6. 版本总结

### 学到了什么

- `MapperFactoryBean` 是 MyBatis-Spring 创建 Mapper Bean 的工厂，继承它并重写 `getObject()` 是注入自定义逻辑的标准方式。
- `@MapperScan(factoryBean = ...)` 可以全局替换所有 Mapper 的工厂类，一行配置覆盖所有 Mapper，无需逐个修改。
- JDK 动态代理的双层嵌套：外层是我们的分片代理，内层是 MyBatis 原始 Mapper 代理，`method.invoke(originalMapper, args)` 触发内层代理执行真实 SQL。
- `BoundSql.getParameterMappings()` 返回 SQL 中每个 `?` 对应的参数映射，通过 `MetaObject` 可以从参数对象中按属性名取值，实现参数的有序展开。

### 与前版本对比

| 项目 | v5.0 | v6.0 |
|------|------|------|
| 分片触发方式 | 手动调用 ShardingContext.set() | Mapper 代理自动触发 |
| 业务代码侵入 | 有（需手动 set） | 无（完全透明） |
| 全链路打通 | 否 | 是 |
| 参数提取 | 手动构造 | 自动从 BoundSql 提取 |

### 局限性

- ThreadLocal 在方法执行后没有清理（内存泄漏，v7.0 修复）。
- `buildParamObject` 的多参数处理是简化版，完整实现需要处理 `@Param` 注解（可参考 MyBatis `ParamNameResolver`）。
- 不支持广播表（v9.0 扩展）和范围查询（v10.0 扩展）。

### 设计要点

v6.0 完成了整个框架的"闭环"：

```
代理层（何时计算）→ Engine（算什么）→ ThreadLocal（怎么传递）
→ 拦截器（改写 SQL）→ DataSource（路由连接）
```

每一层职责单一，通过 `ShardingContext`（ThreadLocal）解耦，任何一层都可以独立替换或扩展，这是整个框架最核心的架构设计。
