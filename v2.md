# v2.0 — 分片上下文：ThreadLocal 传递路由结果

## 1. 版本目标

解决 v1.0 中 `currentKey` 是实例变量、多线程不安全的问题。

引入 `ShardingContext`，用 ThreadLocal 在同一请求线程内传递分片结果，替代硬编码 key。每个线程独立持有自己的路由信息，互不干扰。

---

## 2. 核心功能说明

### 类比

> 给每个快递员（线程）发一张"便利贴"，上面写着"这次去哪个仓库"，全程揣在口袋里。不同快递员的便利贴互不影响，即使同时出发也不会拿错。

### ThreadLocal 工作原理

```
线程 A                          线程 B
  │                               │
  ├─ ShardingContext.set("ds0")   ├─ ShardingContext.set("ds1")
  │                               │
  ├─ 执行 SQL                     ├─ 执行 SQL
  │                               │
  ├─ determineCurrentLookupKey()  ├─ determineCurrentLookupKey()
  │    → 读到 "ds0" ✓             │    → 读到 "ds1" ✓
  │                               │
  └─ 路由到 ds0                   └─ 路由到 ds1
```

每个线程的 ThreadLocal 槽位独立，互不可见。

### 流程图

```
业务代码调用前
  ShardingContext.set(new ShardingResult("ds1", sql))
        │
        ▼  (同一线程内)
MyBatis 执行 SQL
        │
        ▼
ShardingDataSource.determineCurrentLookupKey()
        │  ShardingContext.get().getTargetDataSourceName()
        ▼
路由到 ds1
```

## 4. 代码实现说明

### ShardingResult.java

封装分片计算的结果：目标数据源名称 + 改写后的目标 SQL。

```java
public class ShardingResult {

    // 目标数据源名称，如 "ds0" / "ds1"
    private final String targetDataSourceName;

    // 改写后的物理 SQL，如 "select * from user1 where id=?"
    // v2.0 中暂时不使用，为后续版本预留
    private final String targetSqlStatement;

    public ShardingResult(String targetDataSourceName, String targetSqlStatement) {
        this.targetDataSourceName = targetDataSourceName;
        this.targetSqlStatement = targetSqlStatement;
    }

    public String getTargetDataSourceName() {
        return targetDataSourceName;
    }

    public String getTargetSqlStatement() {
        return targetSqlStatement;
    }

    @Override
    public String toString() {
        return "ShardingResult{ds=" + targetDataSourceName
                + ", sql=" + targetSqlStatement + "}";
    }
}
```

### ShardingContext.java

对 ThreadLocal 的薄封装，提供 get/set 两个静态方法。

```java
public class ShardingContext {

    private static final ThreadLocal<ShardingResult> LOCAL = new ThreadLocal<>();

    public static void set(ShardingResult result) {
        LOCAL.set(result);
    }

    public static ShardingResult get() {
        return LOCAL.get();
    }

    // v7.0 会用到，此处先预留
    public static void remove() {
        LOCAL.remove();
    }
}
```

### ShardingDataSource.java（修改）

`determineCurrentLookupKey()` 改为从 `ShardingContext` 读取，不再依赖实例变量。

```java
public class ShardingDataSource extends AbstractRoutingDataSource {

    @Override
    protected Object determineCurrentLookupKey() {
        ShardingResult result = ShardingContext.get();
        if (result == null) {
            return null; // 返回 null 时使用默认数据源
        }
        String key = result.getTargetDataSourceName();
        System.out.println("determineCurrentLookupKey = " + key);
        return key;
    }
}
```

### ApplicationRunner 验证代码

```java
@Component
public class TestRunner implements ApplicationRunner {

    @Autowired
    private UserMapper userMapper;

    @Override
    public void run(ApplicationArguments args) throws Exception {
        // 手动设置路由到 ds0
        ShardingContext.set(new ShardingResult("ds0", ""));
        List<User> users0 = userMapper.findAll();
        System.out.println("ds0 users: " + users0);

        // 手动设置路由到 ds1
        ShardingContext.set(new ShardingResult("ds1", ""));
        List<User> users1 = userMapper.findAll();
        System.out.println("ds1 users: " + users1);

        // 并发验证：两个线程分别路由到不同数据源
        Thread t1 = new Thread(() -> {
            ShardingContext.set(new ShardingResult("ds0", ""));
            System.out.println("Thread-1 key: "
                + ShardingContext.get().getTargetDataSourceName());
        });
        Thread t2 = new Thread(() -> {
            ShardingContext.set(new ShardingResult("ds1", ""));
            System.out.println("Thread-2 key: "
                + ShardingContext.get().getTargetDataSourceName());
        });
        t1.start(); t2.start();
        t1.join();  t2.join();
    }
}
```

## 6. 版本总结

### 学到了什么

- ThreadLocal 为每个线程提供独立的变量副本，是实现线程安全的"请求上下文"的标准方案。
- 将路由结果封装成 `ShardingResult` 对象，比传递裸字符串更具扩展性（后续可以加 targetSql、role 等字段）。
- `determineCurrentLookupKey()` 返回 `null` 时，`AbstractRoutingDataSource` 会使用 `defaultTargetDataSource`。

### 与前版本对比

| 项目 | v1.0 | v2.0 |
|------|------|------|
| 路由 key 存储 | 实例变量 | ThreadLocal |
| 多线程安全 | 否 | 是 |
| 路由结果封装 | 裸字符串 | ShardingResult 对象 |
| 自动计算路由 | 否 | 否（仍需手动 set） |

### 局限性

- 路由 key 仍需业务代码手动调用 `ShardingContext.set()`，侵入业务层。
- 没有自动根据 SQL 参数计算目标库/表的能力。
- ThreadLocal 用完后没有清理（内存泄漏风险，v7.0 修复）。

### 设计要点

`ShardingResult` 同时携带 `targetDataSourceName` 和 `targetSqlStatement`，这是一个关键设计：数据源路由和 SQL 改写的结果被绑定在同一个对象里，通过同一个 ThreadLocal 传递，保证了同一次请求中两者的一致性。
