# v5.0 — MyBatis 拦截器替换逻辑 SQL

## 1. 版本目标

解决"如何在 SQL 真正发送给数据库之前，把逻辑 SQL 替换成物理 SQL"的问题。

实现 `SqlStatementInterceptor`，作为 MyBatis Plugin 拦截 `StatementHandler.prepare`，从 `ShardingContext` 取出目标 SQL，用 `Unsafe` 替换 `BoundSql` 中的 `sql` 字段。业务代码无感知，SQL 改写在框架层透明完成。

---

## 2. 核心功能说明

### 类比

> SQL 发出去之前，有个"审查员"把快递单上的"逻辑地址"偷偷换成"实际地址"，司机（数据库驱动）只看到真实地址，完全不知道中间发生了什么。

### MyBatis Plugin 拦截机制

```
MyBatis 执行链路：
  Executor → StatementHandler → ParameterHandler → ResultSetHandler

拦截点选择：
  StatementHandler.prepare(Connection, Integer)
  ↑ 此时 BoundSql 已生成，SQL 字符串已确定，但还未发送给数据库
  ↑ 是替换 SQL 的最佳时机
```

### BoundSql.sql 字段替换原理

```
BoundSql 是 final 类，sql 字段没有 setter
        │
        ▼
使用 sun.misc.Unsafe 直接操作内存
  1. Field sqlField = BoundSql.class.getDeclaredField("sql")
  2. long offset = unsafe.objectFieldOffset(sqlField)
  3. unsafe.putObject(boundSql, offset, newSql)
        │
        ▼
BoundSql.sql 字段被替换为物理表 SQL
```

### 完整拦截流程

```
MyBatis 准备执行 SQL
        │
        ▼
SqlStatementInterceptor.intercept(invocation)
        │
        ├─ 取出 StatementHandler
        ├─ 取出 BoundSql（含原始逻辑 SQL）
        ├─ ShardingContext.get() → ShardingResult
        │
        ├─ 若 result != null 且 targetSql != 原始 sql:
        │     Unsafe.putObject(boundSql, sqlFieldOffset, targetSql)
        │     ← BoundSql.sql 字段被替换为物理表 SQL
        │
        ▼
invocation.proceed()  ← 继续执行，数据库收到的是物理 SQL
```

## 4. 代码实现说明

### SqlStatementInterceptor.java

MyBatis Plugin 的标准写法：`@Intercepts` 声明拦截点，实现 `Interceptor` 接口。

```java
@Intercepts({
    @Signature(
        type  = StatementHandler.class,
        method = "prepare",
        args  = {Connection.class, Integer.class}
    )
})
public class SqlStatementInterceptor implements Interceptor {

    // BoundSql.sql 字段的内存偏移量，类加载时计算一次
    private static final Unsafe UNSAFE;
    private static final long SQL_FIELD_OFFSET;

    static {
        try {
            Field f = Unsafe.class.getDeclaredField("theUnsafe");
            f.setAccessible(true);
            UNSAFE = (Unsafe) f.get(null);

            Field sqlField = BoundSql.class.getDeclaredField("sql");
            SQL_FIELD_OFFSET = UNSAFE.objectFieldOffset(sqlField);
        } catch (Exception e) {
            throw new ExceptionInInitializerError(e);
        }
    }

    @Override
    public Object intercept(Invocation invocation) throws Throwable {
        // 取出 StatementHandler（可能被多层代理包装）
        StatementHandler handler = (StatementHandler)
            invocation.getTarget();
        MetaObject metaObject = SystemMetaObject.forObject(handler);
        StatementHandler realHandler = (StatementHandler)
            metaObject.getValue("h.target");  // 解包代理

        // 取出 BoundSql
        BoundSql boundSql = realHandler != null
            ? realHandler.getBoundSql()
            : handler.getBoundSql();

        // 从 ShardingContext 取出分片结果
        ShardingResult result = ShardingContext.get();

        if (result != null) {
            String targetSql = result.getTargetSqlStatement();
            String originalSql = boundSql.getSql();

            // 只有 SQL 确实不同时才替换，避免无谓操作
            if (targetSql != null && !targetSql.isEmpty()
                    && !targetSql.equals(originalSql)) {
                UNSAFE.putObject(boundSql, SQL_FIELD_OFFSET, targetSql);
                System.out.println("SQL replaced: [" + originalSql
                    + "] → [" + targetSql + "]");
            }
        }

        return invocation.proceed();
    }

    @Override
    public Object plugin(Object target) {
        // 只对 StatementHandler 生成代理，减少不必要的代理开销
        if (target instanceof StatementHandler) {
            return Plugin.wrap(target, this);
        }
        return target;
    }
}
```

### ShardingAutoConfiguration.java（新增注册拦截器）

MyBatis 会自动检测 Spring 容器中的 `Interceptor` Bean 并注册，只需声明 Bean 即可。

```java
@Configuration
@EnableConfigurationProperties(ShardingProperties.class)
public class ShardingAutoConfiguration {

    @Bean
    public ShardingDataSource shardingDataSource(ShardingProperties properties) {
        // ... 同前，省略
    }

    @Bean
    public ShardingEngine shardingEngine(ShardingProperties properties) {
        return new StandardShardingEngine(properties);
    }

    // MyBatis 自动识别容器中的 Interceptor Bean 并注册
    @Bean
    public SqlStatementInterceptor sqlStatementInterceptor() {
        return new SqlStatementInterceptor();
    }
}
```

### UserMapper.java（示例）

```java
@Mapper
public interface UserMapper {
    @Select("select * from user where id = #{id}")
    User findById(@Param("id") int id);

    @Insert("insert into user(id, name, age) values(#{id}, #{name}, #{age})")
    void insert(User user);
}
```

### ApplicationRunner 验证代码

```java
@Component
public class TestRunner implements ApplicationRunner {

    @Autowired
    private UserMapper userMapper;

    @Override
    public void run(ApplicationArguments args) throws Exception {
        // 手动设置：路由到 ds0，SQL 替换为 user1 版本
        ShardingContext.set(new ShardingResult("ds0",
            "select * from user1 where id = ?"));

        // userMapper 原始 SQL 是 "select * from user where id = ?"
        // 拦截器会把它替换为 "select * from user1 where id = ?"
        User user = userMapper.findById(1);
        System.out.println("result: " + user);

        // 验证不设置 ShardingContext 时，SQL 不被替换
        ShardingContext.set(null);
        User user2 = userMapper.findById(1);
        System.out.println("no sharding result: " + user2);
    }
}
```

```yaml
    com.example.masharding.mapper: DEBUG
```

```bash
mvn spring-boot:run
```

预期控制台输出：

```

==>  Preparing: select * from user1 where id = ?

<==        Row: 1, alice, 25
```


## 6. 版本总结

### 学到了什么

- MyBatis Plugin 的标准写法：`@Intercepts` + `@Signature` 声明拦截点，`Plugin.wrap()` 生成代理。
- `BoundSql` 的 `sql` 字段是 `private final`，无法通过反射 `set`（因为 `final`），必须用 `Unsafe.putObject()` 绕过 JVM 的访问控制直接写内存。
- `Unsafe.objectFieldOffset()` 在类加载时计算一次并缓存，避免每次拦截都重复计算。
- MyBatis 会自动将 Spring 容器中的 `Interceptor` Bean 注册到 `Configuration`，无需手动调用 `addInterceptors()`。

### 与前版本对比

| 项目 | v4.0 | v5.0 |
|------|------|------|
| SQL 改写时机 | Engine 计算时（字符串操作） | MyBatis 执行前（内存替换） |
| 业务代码侵入 | 需手动调用 engine | 无感知（拦截器透明处理） |
| 改写可靠性 | 正则替换 | 直接替换 BoundSql.sql 字段 |
| 与 MyBatis 集成 | 否 | 是 |

### 局限性

- 使用 `Unsafe` 是 JDK 内部 API，在 JDK 17+ 需要添加 `--add-opens` JVM 参数。
- 拦截器依赖 `ShardingContext` 中已有正确的 `targetSqlStatement`，若 Engine 没有提前计算，SQL 不会被替换。
- 此版本仍需手动调用 `ShardingContext.set()`，v6.0 会通过 Mapper 代理自动完成。

### 设计要点

选择拦截 `StatementHandler.prepare` 而非 `Executor.query/update` 的原因：`prepare` 阶段 `BoundSql` 已经生成（参数已绑定），但 `PreparedStatement` 还未创建，是替换 SQL 字符串的最后时机，也是最干净的切入点。
